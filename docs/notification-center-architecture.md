# 通知中心与强化日记通知架构归档

## 文档定位

本文记录 siyuan-homepage 当前通知系统的最终架构、职责边界、业务接入情况、VIP 规则、移动计划策略、数据可靠性约束和后续维护规则。本文描述的是当前有效架构，不把开发过程中的过渡方案、已删除入口或已废弃实现视为现状。

通知能力必须始终遵守以下第一性原则：

- 不破坏思源本体，不修改或回滚业务数据。
- 不污染用户主题，不影响其他插件的样式、数据和运行行为。
- 不覆盖用户已有的通知设置、业务规则、Webhook、飞书地址、Header、密钥、通知历史和移动计划。
- 不因读取失败、目录异常或文件损坏而把真实数据解释为空数据。
- 不为通知新增正文 `LIKE` 全库扫描；通知只复用业务模块已有的安全数据源和索引。
- 通知投递失败不能回滚已经成功完成的任务、日记、复习、番茄钟或其他业务操作。
- 所有通知能力必须可关闭、可解释和可恢复；错误必须保留足够信息供用户判断和后续重试。

## 统一通知架构

### 业务模块职责

业务模块只负责决定“为什么通知”和“通知什么”，包括：

- 通知条件与业务规则。
- 通知标题和正文。
- 计划通知时间、有效期和业务发生实例。
- 投递目标选择。
- 业务数据读取及其错误隔离。

业务模块向统一通知中心提交标准通知事件，不自行实现投递基础设施。

### notification-center 职责

`notification-center` 是唯一的通知基础设施层，统一负责：

- 桌面系统通知。
- Android、iOS、鸿蒙的移动端即时通知。
- 当前移动设备的本地通知计划。
- Webhook 和飞书投递。
- 通知历史。
- 发生实例去重和投递去重。
- 发送频率限制。
- 设备识别及桌面、普通浏览器、移动容器的环境判断。
- VIP 能力判断和底层权限校验。

业务模块禁止直接执行以下操作：

- 直接调用 `new Notification(...)`。
- 直接调用 `platformUtils.sendNotification(...)`。
- 直接请求 Webhook 或飞书接口。
- 自行建立另一套通知历史、去重、限流或移动计划。

浏览器 Notification API 和思源移动通知 API 只能出现在 `notification-center` 的通道实现中。业务统一通过 `notificationCenter.notify` 接入，通知失败只形成可解释的投递结果和历史，不改变业务事务结果。

## VIP 边界

以下能力全部为 VIP 专属：

- 桌面系统通知。
- Android、iOS、鸿蒙通知。
- 移动通知计划。
- Webhook。
- 飞书。
- 通知中心配置。
- 各业务通知配置。

免费用户仍保留普通思源 `showMessage` 操作反馈，以及插件原有的笔记内部随机会员推广弹窗；二者不属于统一通知中心的系统或远程通知能力。免费用户不能通过旧入口、兼容层或业务模块旁路调用系统通知、移动通知、Webhook 或飞书。

会员失效时保留通知中心设置、业务规则和通知历史，不清空或重写用户配置；当前设备的移动通知计划进入撤销流程。单条取消失败时必须保留对应本地计划记录和错误信息，后续继续重试，不能因一个失效 `notificationId` 删除其他有效计划。会员重新开通后，根据原有设置和业务数据重新对账并恢复当前设备计划。

VIP 校验同时存在于配置保存边界和实际通知调用边界。仅隐藏 UI 不能替代底层校验，嵌入其他页面也不能绕过权限。

## 业务接入清单

### 任务通知

任务通知的唯一 UI 配置入口位于强化日记工作台。其边界固定为：

- 唯一配置文件：`taskNotifySettings.json`。
- 唯一执行模块：`task-notify`。
- 唯一调度器：`taskNotifyScheduler`。
- 唯一移动 Provider：`taskMobileNotificationPlanProvider`。

任务管理 Plus 不提供通知配置入口，移动任务组件也不提供通知配置入口。它们不能复制任务通知规则、启动第二个任务通知调度器或注册第二个任务移动 Provider。

强化日记工作台复用现有任务通知设置表单，直接管理同一份 `taskNotifySettings.json`。任务通知仍由 `task-notify` 唯一读取、调度和发送，不要求页面中同时存在任务管理 Plus 组件。

### 强化日记通知

强化日记统一通知面板按以下职责组织：

1. 通用任务通知。
2. 日记与复盘。
3. 任务整理。
4. 项目进展。
5. 数据维护。

通用任务通知管理 `taskNotifySettings.json`；其余强化日记专属通知管理 `enhancedDiaryNotifySettings.json`。两个文件保持独立加载、独立保存和独立错误状态，不能合并成新的配置协议，也不能用一个文件的默认值覆盖另一个文件的异常内容。

项目通知属于强化日记工作台，没有 `project-notify`、`projectNotifySettings.json`、独立项目调度器或独立项目移动 Provider。

### 其他业务

- 纪念日通知继续由 `countdown-notify` 负责。
- 复习文档通知继续由 `review-notify` 负责。
- 番茄钟通知继续由 `focus-notify` 负责，并通过统一通知中心投递，不直接调用浏览器 Notification。
- AI、MCP、Agent 当前未接入通知系统；未来如有产品需求必须单独立项。

## 强化日记通知结构

### 通用任务通知

- 任务指定时间提醒。
- 今日任务摘要。
- 明日任务摘要。
- 逾期任务摘要。
- 高优先级任务摘要。
- 自定义筛选任务摘要。

这些规则全部复用 `task-notify` 的既有规则类型、字段、筛选逻辑、正文、调度行为和移动计划，不在强化日记配置中复制数据。

### 日记与复盘

- 今日日记未创建。
- 昨日未复盘。
- 日复盘。
- 周复盘。
- 月复盘。
- 年度复盘。

### 任务整理

- 未迁移任务。
- 工作台逾期任务。
- 长期未处理任务。

### 项目进展

- 项目逾期风险。
- 项目长期停滞。
- 项目完成待归档。
- 项目周进展。

### 数据维护

- 模板缺失。
- 项目关系异常。
- 项目索引异常。

数据维护信息继续优先留在强化日记内部通知面板，由强化日记既有数据维护能力解释和处理，不拆分为独立项目通知模块。

## 移动通知计划策略

### 可以提前注册的提醒

- 任务固定时间提醒。
- 纪念日固定提醒。
- 复习到期提醒。
- 强化日记固定周、月、年复盘提醒。

这些提醒在当前设备、当前 VIP 状态、移动通知设置和规划时间范围均有效时，可以注册为系统本地通知计划，使思源不在前台时仍可由系统触发。

### 不提前固化的动态摘要

- 今日任务动态摘要。
- 项目风险。
- 项目停滞。
- 项目周进展。
- 任务整理。
- 实时日记状态。

动态摘要依赖通知时刻附近的最新业务状态，需要思源在通知时间附近运行。不能把旧快照长期固化为移动计划，也不能为了提前计划而引入新的全库扫描。

### 安全替换和对账

同一设备的所有实际计划修改必须串行执行。计划替换遵守以下顺序：

1. 注册新通知。
2. 保存新 `notificationId` 并读回验证。
3. 取消旧通知。

任何一步失败都以保留旧计划、避免漏提醒为优先，不允许先删除旧记录再尝试创建新通知。Provider 生成失败时保留该来源已有计划；取消失败时保留可重试记录。普通对账和强制重建不能并行写入，同一轮等待结束后必须处理已经合并的尾随请求。

移动计划只在移动运行环境读取和维护；桌面端和普通浏览器不为桌面设备创建或校验移动计划文件。

## 可靠性规则

### 设置和数据文件

- 文件经确认不存在时，加载器返回默认关闭设置或标准空文件。
- 读取 API 失败但无法确认文件不存在时，必须抛出读取错误。
- 真实存在的空文件、非法 JSON、Schema 错误、Version 错误或设备不匹配必须报错。
- 损坏文件不能自动归一化为空数据，不能自动覆盖，界面必须阻止用默认值保存回异常文件。
- 通知中心设置、通知历史和移动计划是独立数据域；一个辅助文件异常不能使其他区域失效。

### Provider 和规则

- 一个 Provider 失败不阻止其他 Provider 生成计划。
- 失败 Provider 对应的已有移动计划必须保留，避免把读取失败误判为“该来源已经没有提醒”。
- 单条规则失败不阻止同模块的后续规则执行。
- 失败规则不能写入“已发送”状态，不能生成成功历史，后续仍可重试。
- 单条移动通知取消失败不终止其他计划的新增、替换和清理；失败记录继续保留。

### 强化日记数据隔离

强化日记的日记、任务、项目数据分别加载和处理。项目数据失败不能阻止日记提醒、复盘提醒或通用任务通知；任务数据失败也不能覆盖日记或项目数据。读取失败是未知状态，不得解释为业务对象不存在或规则已满足。

### 索引和查询

通知只复用强化日记及各业务已经建立的安全索引和结构化读取能力。禁止为了通知新增正文 `LIKE` 全库扫描，禁止通过宽泛 SQL 猜测日记、任务、项目或复习状态。

## 日期、周期和去重规则

所有业务日期使用本地年月日计算和比较。禁止用 `toISOString().slice(0, 10)` 代替本地业务日期，以免时区偏移改变任务日期、日记日期或复盘周期。

复盘周期必须遵守：

- 月末提醒对应当前月周期。
- 次月首日提醒对应上一月周期。
- 年末提醒对应当前年度周期。
- 次年首日提醒对应上一年度周期。

运行时通知与移动计划对同一业务发生实例必须使用相同的 `occurrenceKey`、周期范围和稳定 `planKey` 映射。移动计划不能另建一套去重键；同一规则、同一周期只能生成一个计划。

昨日复盘状态语义固定为：

- `completed`：已完成，不提醒。
- `skipped`：已跳过，属于已处理终态，不提醒。
- `pending`：存在可识别结构但未完成，提醒。
- `missing_template`：缺少可识别复盘结构，提醒。
- 文档检查未知、Markdown 读取失败或读取抛错：不误报，不创建通知历史。

## 数据文件清单

| 数据域 | 文件 | 说明 |
| --- | --- | --- |
| 通知中心设置 | `notification-center/settings.json` | 桌面、移动、Webhook、飞书及迁移状态等统一设置 |
| 通知历史索引 | `notification-center/history-index.json` | 年度历史文件索引 |
| 通知历史 | `notification-center/history-YYYY.json` | 按年份保存统一投递历史 |
| 移动计划 | `notification-center/mobile-plans/<device-id>.json` | 当前设备的本地通知计划和 `notificationId` |
| 任务通知 | `taskNotifySettings.json` | 通用任务通知唯一配置 |
| 纪念日通知 | `countdownNotifySettings.json` | 纪念日规则配置 |
| 强化日记通知 | `enhancedDiaryNotifySettings.json` | 日记、复盘、任务整理和项目进展通知配置 |
| 复习通知 | `reviewNotifySettings.json` | 复习文档通知配置 |
| 番茄钟通知 | `focusNotifySettings.json` | 番茄钟通知配置 |

上述文件的 Schema 和 Version 不得随意修改。新增字段必须提供向后兼容的 normalize，并保留旧用户设置；任何迁移都必须先安全读取、确认目标文件确实不存在或版本允许迁移，不能以读取异常作为覆盖依据。
