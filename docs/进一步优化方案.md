# 强化日记工作台后续整体优化方案

## 一、当前整体判断

现在已经完成的核心能力：

1. 首页工作台标签页已经有完整布局；
2. 任务中心已经支持搜索、筛选、排序、任务详情；
3. 项目中心已经支持项目列表、项目详情、项目任务聚合；
4. 记录中心已经支持今日记录、历史记录、分类、搜索、记录详情；
5. 复盘中心已经支持当前复盘和历史复盘档案；
6. 通知中心已经支持筛选、详情和操作入口；
7. 日历已经支持月视图和日期详情；
8. 写入逻辑基本坚持了 Markdown 标题契约和 Tasks Plus 格式；
9. 复杂业务内容没有放进插件 JSON，这个方向是对的。

现在主要问题不是“缺功能”，而是：

> 功能分散，导航层级像后台系统，但总览页还没有真正承担“个人中控台”的作用。

也就是说，当前各个中心都有，但它们之间的联动还不够。用户进入工作台以后，最自然的路径应该是：

1. 先看今天整体状态；
2. 看今天日历/时间线；
3. 处理最紧急任务和提醒；
4. 快速记录或新增任务；
5. 再进入任务、项目、记录、复盘等细分中心。

现在的总览页还偏“摘要卡片”，日历单独放在一个标签页，通知也单独放在一个标签页，所以工作台有点像多个页面拼在一起，而不是一个真正的工作中枢。

---

# 二、信息架构建议

## 1. 侧边栏结构重新调整

现在侧边栏是：

* 总览
* 任务
* 项目
* 记录
* 日历
* 通知
* 复盘

我建议改成：

* 总览
* 任务
* 项目
* 记录
* 复盘
* 更多

其中“日历”和“通知”不要再作为完全独立的一等标签，而是融入总览页。

具体方案：

### 总览页内置

总览页放：

* 今日概览；
* 今日任务；
* 逾期与提醒；
* 快速记录；
* 项目推进；
* 日历概览；
* 今日复盘状态。

### 通知中心降级为模块

通知中心可以从：

* 总览页的“提醒中心”模块进入；
* 顶部通知 badge 进入；
* 侧边栏“更多”里进入。

### 日历中心降级为模块

日历可以从：

* 总览页下半区直接展示；
* “查看完整日历”按钮打开大面板；
* 或保留日历标签，但不作为主要入口。

我更推荐：**总览页嵌入日历，日历标签可以暂时保留，但改成“日历详情”或“更多 > 日历”。**

这样更符合你说的：“日历可以不放在单独的标签页，放在总览页目前几个块下面。”

# 三、总览页应该升级成真正的 Dashboard

总览页现在应该是整个工作台的核心，而不是一个普通摘要页。

建议总览页改成下面结构。

## 1. 顶部状态区

顶部仍然保留当前统计卡，但要更有业务含义：

* 今日待办；
* 逾期任务；
* 今日记录；
* 今日推进项目；
* 待处理提醒；
* 复盘完成度。

这些卡片点击后不只是跳到对应标签，而应该带着筛选条件跳过去。例如：

* 点击“逾期任务” → 进入任务中心，并自动筛选为逾期；
* 点击“今日记录” → 进入记录中心，并自动选中今日记录；
* 点击“待处理提醒” → 打开总览页提醒模块或通知详情；
* 点击“复盘完成度” → 进入复盘中心当前复盘。

目前只是切换 tab，不带上下文，后续可以优化。

## 2. 今日行动区

建议在总览页最上方放一个“今日行动”模块，而不是让用户自己去各个中心找。

内容包括：

* 今日必须处理；
* 逾期任务；
* 建议迁移；
* 当前复盘；
* 模板缺失；
* 今日无日记提醒。

这个模块应该以“待办行动列表”的方式呈现，每条都有按钮：

* 完成；
* 迁移；
* 补模板；
* 打开日记；
* 标记复盘完成。

它相当于通知中心的精选版。

## 3. 今日时间线

总览页可以加一个“今日时间线”模块，把今天的内容按时间或类别串起来：

* 13:20 快速记录；
* 新建任务；
* 完成任务；
* 项目推进；
* 复盘标记；
* 任务迁移。

当前你已经有：

* `## 新建任务`
* `## 迁移任务`
* `## 快速记录`
* `## 项目推进`
* `## 任务动态`

这些都可以被汇总成今日时间线。第一版不需要真的精确到所有块创建时间，可以先按区块顺序展示。

## 4. 日历嵌入总览页

把日历月视图移到总览页下半区，结构可以是：

左侧：月历
右侧：选中日期详情
下方：本周/本月统计

总览页里的日历不必占满全页，但要能直接完成：

* 看哪天有日记；
* 看哪天有记录；
* 看哪天有任务；
* 看哪天复盘完成；
* 点击某天查看详情；
* 打开当天日记。

这样日历就不是单独的“查看页”，而是总览页的数据入口。

## 5. 今日快速入口

总览页要有更明显的快捷操作：

* 新建任务；
* 快速记录；
* 打开今日日记；
* 补今日模板；
* 标记今日复盘完成；
* 迁移旧任务。

现在这些功能分散在 header、任务页、记录页、通知页里，建议在总览页聚合。

# 四、日历功能可以继续丰富

日历现在已经有月视图和日期详情，但还可以变成很专业的“生活/工作热力图”。

## 1. 日历上显示多维状态

每一天可以显示：

* 日记是否存在；
* 快速记录数量；
* 新建任务数量；
* 完成任务数量；
* 逾期任务数量；
* 复盘完成状态；
* 项目推进数量。

不建议在格子里塞太多文字，应该用小点/小条/数字组合。

例如：

* 绿色点：有日记；
* 蓝色点：有记录；
* 橙色点：有任务；
* 红色点：有逾期；
* 勾号：复盘完成。

## 2. 日历支持视图切换

可以加：

* 月视图；
* 周视图；
* 列表视图；
* 热力图视图。

第一版优先：

* 月视图；
* 最近 30 天列表；
* 本月热力图。

## 3. 日期详情更丰富

当前日期详情可以进一步显示：

* 这天的日记状态；
* 这天的新建任务；
* 这天迁移来的任务；
* 这天快速记录；
* 这天项目推进；
* 这天复盘状态；
* 打开日记；
* 补模板；
* 创建今天日记，仅今天允许；
* 查看这天完整记录。

非今天日期只读，今天允许写入。

## 4. 日历和记录/复盘联动

点击某天：

* 可以切换记录中心并筛选该日期；
* 可以切换复盘中心并定位该日；
* 可以打开当天日记。

这个联动会让工作台从“多个页面”变成“统一数据系统”。

# 五、任务中心继续增强方向

任务中心已经有列表和详情，但还可以更专业。

## 1. 任务视图模式

建议增加三种视图：

* 列表视图；
* 看板视图；
* 时间视图。

### 列表视图

当前已经有。

### 看板视图

按状态分：

* 今日；
* 逾期；
* 等待；
* 已完成；
* 建议迁移。

不一定要拖拽，第一版只读分组即可。

### 时间视图

按日期分：

* 今天；
* 明天；
* 本周；
* 未来；
* 无日期。

这对于任务管理很实用。

## 2. 批量操作

任务多以后，单条操作会麻烦。可以增加批量选择：

* 批量完成；
* 批量迁移到今天；
* 批量推迟到明天；
* 批量删除；
* 批量添加标签。

要注意，批量删除和批量迁移必须二次确认。

## 3. 任务推迟功能

现在有迁移到今天，但还缺“推迟”。

可以加：

* 推迟到明天；
* 推迟到下周；
* 修改截止日期；
* 修改开始日期。

它本质上是修改 Tasks Plus 任务行里的 `⌛` 和 `📅`。

## 4. 任务和记录互转

这个很有价值：

* 记录转任务；
* 任务添加记录；
* 任务完成后写入任务动态。

例如快速记录里某条“问题：需要修复数据库写入”，可以转成：

```markdown
- [ ] 修复数据库写入 ❗❗ ⌛2026-05-21 📅2026-05-22 #插件#
```

写入今天的 `## 新建任务`。

## 5. 任务详情补充“操作历史”

任务详情里可以显示：

* 来源日记；
* 迁移来源；
* 最近修改；
* 相关任务动态；
* 相关快速记录。

这需要扫描 `## 任务动态` 和迁移子内容。

---

# 六、项目中心继续增强方向

项目中心现在是只读聚合，这是对的。下一步可以增强“项目工作台感”。

## 1. 项目详情页更像项目档案

每个项目详情可以分区：

* 项目概览；
* 未完成任务；
* 今日任务；
* 逾期任务；
* 最近记录；
* 项目推进；
* 下步计划；
* 历史进展。

项目本身不需要数据库，仍然通过：

* Tasks Plus 标签；
* `## 项目推进` 下的三级标题；
* 快速记录内容；
* 任务动态；

来聚合。

## 2. 项目时间线

项目详情可以有时间线：

* 哪天有项目推进；
* 哪天创建任务；
* 哪天完成任务；
* 哪天有相关记录。

这会非常有用。

## 3. 项目筛选增强

可以支持：

* 只看有逾期任务的项目；
* 只看今天有推进的项目；
* 只看长期未推进的项目；
* 只看已完成项目；
* 只看活跃项目。

其中“长期未推进”很实用。例如 7 天没有项目推进，但还有未完成任务，就提示。

## 4. 项目主页入口

可以提供一个按钮：

* “生成今日项目推进模板”
* “打开项目相关日记”
* “查看项目历史”

不需要创建项目数据库。

---

# 七、记录中心继续增强方向

记录中心现在已经支持历史检索，后面可以向“个人知识流”发展。

## 1. 记录时间线

当前按分类筛选，但可以增加时间线视图：

* 今天；
* 昨天；
* 本周；
* 本月。

每条记录显示：

* 日期；
* 分类；
* 内容摘要；
* 来源日记；
* 操作。

## 2. 记录类型增强

现在分类是：

* 未分类；
* 想法；
* 问题；
* 决策；
* 日志。

可以增加：

* 灵感；
* 会议；
* 学习；
* Bug；
* 读书；
* 实验；
* 论文。

但不建议写死太多。更好的方式是：

* 默认保留五类；
* 允许用户在模板中新增三级标题分类；
* 记录中心自动识别 `## 快速记录` 下所有三级标题作为分类。

现在如果只识别固定五类，用户扩展性不够。这个值得优化。

## 3. 记录转任务 / 记录绑定项目

记录详情可以加：

* 转为任务；
* 复制为任务；
* 关联项目标签；
* 添加到今日任务动态。

不强制写隐藏属性，仍用 Markdown。

## 4. 记录全文搜索

当前是最近 90 天扫描，后续可以做：

* 最近 7 / 30 / 90 / 180 天；
* 全部日记搜索；
* 只搜索标题；
* 搜索正文。

全量搜索可能性能较重，建议加上“点击搜索全部”而不是默认加载。

---

# 八、复盘中心继续增强方向

复盘中心已经有当前复盘和历史档案，可以继续做“复盘质量提升”。

## 1. 复盘草稿生成

不是 AI 也可以做：

* 自动列出本周完成任务；
* 自动列出本周未完成任务；
* 自动列出本周记录；
* 自动列出项目推进；
* 插入到周记/月记模板对应区域。

这可以先做“生成草稿”，不是自动覆盖。

例如：

按钮：生成本周复盘草稿

写入：

```markdown
## 本周完成

- [x] ...

## 本周未完成

- [ ] ...

## 本周重要记录

- 想法：...
- 决策：...
```

必须注意：只在对应区块为空或用户确认后插入，不覆盖已有内容。

## 2. 复盘完成度评分

可以基于模板区块判断：

* 是否有完成标记；
* 是否填写了“发生了什么”；
* 是否填写了“收获”；
* 是否填写了“下一步”。

给出一个复盘质量评分，而不是只看完成标记。

## 3. 复盘连续性

可以显示：

* 连续记录天数；
* 本周复盘是否完成；
* 本月复盘是否完成；
* 最近 30 天日记完成率；
* 最近 12 周周记完成率。

这个适合放总览页。

---

# 九、通知中心继续增强方向

通知中心现在可以筛选和详情，后续要做的是“处理策略”。

## 1. 通知分级处理

可以把提醒分成：

* 必须处理；
* 建议处理；
* 信息提示。

当前 danger/warning/info 已有基础。后续可以给总览页只显示“必须处理”。

## 2. 通知暂不处理

可以加：

* 今天不再提醒；
* 本周不再提醒；
* 忽略此任务迁移提醒；
* 忽略此复盘周期。

这个需要保存状态。保存位置要谨慎：

* 临时 UI 状态可以放组件内；
* 真正长期忽略需要写入插件配置；
* 对具体任务忽略最好不要用块属性，可能需要一个轻量 JSON 状态文件。

建议第一版只做“本次会话隐藏”，不持久化。

## 3. 通知转行动

通知中心每条提醒应该有清晰动作：

* 逾期任务 → 打开任务详情 / 推迟 / 完成；
* 迁移建议 → 迁移到今天 / 暂不迁移；
* 模板缺失 → 补模板；
* 复盘提醒 → 打开 / 标记完成 / 跳过。

有些动作现在已有，有些还可以补。

# 十、专业化交互建议

## 1. 全局命令面板

可以在工作台加一个“命令面板”，快捷执行：

* 新建任务；
* 快速记录；
* 打开今日日记；
* 搜索任务；
* 搜索记录；
* 跳转项目；
* 打开复盘；
* 打开日历日期。

类似 `Ctrl/Cmd + K` 的工作台内部搜索。

这会让组件很专业。

## 2. 全局搜索

在工作台顶部加一个搜索框，搜索范围包括：

* 任务；
* 项目；
* 记录；
* 复盘标题；
* 通知。

搜索结果按类型分组。点击结果跳到对应中心并选中详情。

这是后续最值得做的功能之一。

## 3. 快捷创建浮层

现在新建任务、快速记录分散。可以做一个右下角或 header 的“快速添加”按钮：

* 新建任务；
* 快速记录；
* 新建项目推进；
* 写任务动态；
* 打开今日复盘。

## 4. 上下文跳转

每个对象都应该能跳转到相关中心：

* 任务详情 → 项目详情；
* 任务详情 → 来源日记；
* 记录详情 → 相关项目；
* 项目详情 → 任务中心筛选该项目；
* 日历详情 → 记录中心筛选该日期；
* 通知详情 → 对应任务/复盘详情。

这会显著提升专业感。

# 十一、技术架构需要整理的点

现在功能越来越多，代码会越来越大。建议开始做结构整理。

## 1. 抽公共 UI 组件

目前这些组件内部会重复实现：

* toolbar；
* filter select；
* stat card；
* list item；
* detail panel；
* badge；
* empty state；
* action buttons。

建议新增：

```text
workspace/components/common/
  WorkspaceToolbar.svelte
  WorkspaceBadge.svelte
  WorkspaceDetailPanel.svelte
  WorkspaceListPanel.svelte
  WorkspaceActionButton.svelte
  WorkspaceSectionCard.svelte
```

这样后面不会每个中心都写一套 CSS。

## 2. 抽统一日期工具

现在多个文件都有日期格式化、范围计算、parseLocalDate。建议新增：

```text
workspace/enhancedDiaryWorkspaceDate.ts
```

包含：

* `formatLocalDate`
* `parseLocalDate`
* `startOfLocalDay`
* `addDays`
* `daysBetween`
* `getDateRange`
* `isSameLocalDate`

避免再次出现 UTC 日期问题。

## 3. 数据加载要懒加载

现在 `loadEnhancedDiaryWorkspaceState()` 会在工作台刷新时加载：

* 任务；
* 今日记录；
* 历史记录 90 天；
* 项目；
* 当前复盘；
* 复盘历史；
* 通知；
* 日历另行加载。

这会越来越重。建议改成：

### 基础 state

打开工作台时只加载：

* 今日摘要；
* 任务摘要；
* 通知；
* 当前复盘。

### Tab 懒加载

进入记录中心时再加载历史记录；
进入复盘中心时再加载复盘历史；
进入日历区域时再加载日历；
进入项目中心时再加载项目详情聚合。

这样工作台打开会更快。

## 4. 加简单缓存

同一天的数据不必每次切 tab 都重新扫。

可以加一个轻量缓存：

```ts
WorkspaceCache {
  tasksByDate
  recordsByRange
  reviewHistoryByDate
  calendarByMonth
}
```

刷新按钮清空缓存。

## 5. 错误边界

某个中心加载失败不应该让整个工作台失败。

比如历史记录扫描失败，只显示记录中心错误；
复盘历史失败，只显示复盘中心错误；
日历失败，只显示日历错误。

现在一些失败会 fallback，但可以更系统。

# 十二、模板和兼容性优化

## 1. 模板结构检测页

现在模板缺失会提示，但用户不一定知道缺了什么。可以做一个“模板诊断”入口：

显示：

* 今日日记区是否存在；
* 新建任务区是否存在；
* 迁移任务区是否存在；
* 快速记录区是否存在；
* 项目推进区是否存在；
* 任务动态区是否存在；
* 今日复盘区是否存在；
* 快速记录分类是否完整。

每项有：

* 状态；
* 说明；
* 一键补充缺失区块。

这个很实用。

## 2. 模板升级向导

因为用户可能已有旧模板，可以做：

* 检测旧模板；
* 显示缺失结构；
* 一键追加工作台结构；
* 不覆盖已有内容。

## 3. 支持自定义记录分类

前面说过，快速记录分类最好从模板标题自动读取，而不是固定五类。

# 十五、当前可能存在的具体问题清单

这些不是阻塞，但值得在后续让 Codex 检查：

1. **工作台首次加载可能越来越慢**
   因为历史记录、复盘历史、任务、项目、通知都在一次 state 里加载。

2. **记录分类仍可能固定过死**
   用户如果在模板里新增分类，记录中心未必能自动识别。

3. **任务打开块可能不是真正定位块**
   当前 `onOpenBlock={openDoc}`，可能只是打开文档，不一定定位到块。

4. **统计卡点击没有筛选上下文**
   点击逾期任务只是进入任务中心，不一定自动筛选逾期。

5. **通知没有隐藏/暂不处理能力**
   每次刷新可能还是显示同样提醒。

6. **日历作为独立标签存在感偏弱**
   更适合嵌入总览。

7. **复盘历史可能加载偏重**
   30 天日记 + 12 周 + 12 月 + 5 年，每次刷新都算，后续可懒加载。

8. **多个中心 CSS 重复较多**
   后续应抽 common 组件。

9. **没有全局搜索**
   工作台数据多以后会很需要。

10. **没有模板诊断页**
    用户模板结构缺失时，只看到提示，不知道如何修复。
