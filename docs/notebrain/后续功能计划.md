# Agent Workbench 后续功能规划

## 一、总原则

后续所有新功能都必须继续遵守当前已经确定的第一铁律：

**Workbench 是通用 Agent Harness，不是业务流程控制器。Planner 才是唯一业务决策者。Tool 是全局独立能力，只负责校验参数、执行动作、返回 JSON observation。Skill 是中文能力说明包，不能拥有工具、绑定工具、规定固定步骤。**

所以后续功能不要写死成：

```text
用户问 A → 代码自动搜 B → 代码自动读 C → 代码自动回答
```

而应该设计成：

```text
新增独立 Tool / 独立 Skill / 独立 Tool Provider
Planner 根据用户请求和 Skill 说明自主决定是否调用
```

---

# 二、最高优先级：P0 基础体验稳定

这些是新功能前必须优先完成或持续打磨的能力。它们决定 Agent Workbench 是否“能稳定长期使用”。

## P0-1. 历史对话 JSON 上下文

### 目标

解决连续追问时 AI 丢上下文的问题。

### 设计方向

给 Planner 提供一个轻量、结构化的 `conversationContext` JSON，而不是把完整历史消息、完整工具结果塞进去。

### 上下文应包含

```json
{
  "recentTurns": [
    {
      "turnIndex": 1,
      "createdAt": 1780654125686,
      "user": {
        "text": "上一轮用户问题"
      },
      "assistant": {
        "finalAnswer": "上一轮最终回答",
        "references": [
          {
            "sourceType": "siyuan_doc",
            "docId": "xxx",
            "title": "文档标题"
          }
        ],
        "scope": {
          "type": "whole_kb"
        }
      }
    }
  ],
  "compressed": {
    "summary": "更早历史的压缩摘要"
  }
}
```

### 不应包含

* 历史完整工具 observation；
* 历史 ToolDispatch / ToolResult；
* 完整参考资料正文；
* debug trace；
* full prompt；
* internal path / realPath / internalMapping；
* workbenchEvents。

### 关键规则

历史上下文只让 AI 知道“前面聊到了什么、引用过什么资料”。如果下一轮需要旧资料正文，Planner 应根据 references 里的 `docId / blockId / resourceId` 主动调用 `read_docs`。

---

## P0-2. 读取工具命名为 `read_docs`

### 原因

旧读取工具名带有“搜索候选之后再读取”的流程暗示，会让读取工具看起来依附于搜索工具。

### 新定位

`read_docs` 是全局独立读取工具。Planner 可以根据任意来源调用它：

* 搜索结果里的 docId；
* 文档树里的 docId；
* 历史 references 里的 docId；
* 用户直接给出的 docId；
* 未来文件、网页、MCP 资源返回的 resourceId。

### 建议名称

```text
工具名：read_docs
标题：读取文档
说明：根据 docId、blockId 或 cursor 读取文档/块正文内容。
```

### 相关文案同步

`search_scope`：

```text
搜索结果只是候选，不代表已读取正文；docId 可传给 read_docs。
```

`list_knowledge_map`：

```text
结构结果只表示资料位置，不是正文；docId 可传给 read_docs。
```

---

## P0-3. 引用可信、可复查、可继续读取

### 目标

回答必须让用户知道来源在哪里，并能继续追问。

### 需要加强

* 回答 references 可点击跳转到思源文档或块；
* 区分本地知识库引用、文件引用、网页引用、MCP/API 引用；
* 回答中不要把搜索候选当正文证据；
* 历史上下文保留 references 元信息；
* 用户追问“刚才那篇文档继续展开”时，Planner 能根据历史 references 再调用 `read_docs`。

### 付费价值

这是普通 AI 产品难以替代的核心优势。

---

## P0-4. Debug 默认静默、单入口、内存有界

### 当前方向

保留：

```js
window.__kbAgentDebug()
```

作为唯一控制台调试入口。

### 要求

* 默认不自动 console 打印；
* 不自动复制剪贴板；
* 不暴露多个 `__kbAgent*` 全局变量；
* 只保存最近 3 次 turn trace；
* 生命周期 debug event 最多 150–200 条；
* schema sanity 只保留最近一次；
* debug 不进入 Planner prompt；
* debug 不阻断运行。

### 目的

发布后可以继续保留调试工具，方便后续迭代，不影响普通用户日常使用。

# 三、第一批 VIP 高感知功能：P1

这些是最容易让用户觉得“值得付费”的功能。

## P1-1. 日程 / 任务 / 日记综合 Skill

### 价值

很适合主页插件。用户每天打开主页，就可以看到待办、复盘、日记总结。

### 推荐形态

不能只有 Skill，还需要任务/日记相关工具。

### 可能工具

```text
list_daily_notes
search_tasks
create_task
update_task_status
summarize_day
plan_day
review_week
```

### 场景

* 今天要做什么；
* 昨天遗留了什么；
* 本周完成了什么；
* 哪些任务长期拖延；
* 从日记中提取行动项；
* 根据项目笔记生成今日计划。

### 第一阶段边界

* **第一阶段只做只读 Skill**。Planner 根据用户请求自主搜索、读取日记、任务、待办、复盘内容；不写死“今天/昨天/本周”固定流程，不替用户做时间范围判断。
* **优先复用现有思源知识库工具**。日记、任务、待办在思源中本身就是文档或块，Planner 可通过 `search_scope` / `read_docs` / `list_knowledge_map` 自主查找，不需要为每个时间窗口写死独立工具调用链。
* **写操作放到后续阶段**。创建任务、修改任务状态、写入日记都属于写操作，必须等 `requiresConfirmation` 确认机制稳定后再开放。

---

## P1-2. 联网搜索

### 价值

用户会天然期待 AI 能联网，但你的产品差异点应该是：

> 本地知识库优先，互联网补充验证。

### 推荐工具

```text
web_search
web_read_page
```

### 使用原则

* 只有用户明确要求联网，或开启联网并表达需要最新信息时，Planner 才使用联网工具；
* 本地知识库是第一参考来源；
* 互联网是补充资料；
* 如果互联网信息和本地知识库冲突，必须明确说明差异；
* references 中区分 `web_page` 和 `siyuan_doc`。

### 第一阶段边界

* **作为独立 Web Tool Provider 接入**。`web_search` / `web_read_page` 是独立工具，不和 `search_scope` / `read_docs` 串成固定流程。Planner 根据用户请求和 Skill 说明自主决定是否调用。
* **本地知识库优先，联网搜索补充最新信息**。不要默认开启联网，不要在没有用户明确需求时自动调用。
* **网页引用 sourceType 使用 `web_page`**，和思源文档引用清晰区分。

### 冲突说明示例

```json
{
  "localClaim": "本地笔记中记录 A 方案适合当前实验。",
  "webClaim": "最新网页资料显示 B 方案已有更新。",
  "note": "本地资料和互联网资料存在差异，建议用户确认时间和来源。"
}
```

### 优先级

**P1 第二优先级**。它能满足用户对“最新信息”的期待，但必须避免喧宾夺主。

---

## P1-3. 当前主题整理

### 价值

这是知识库 Agent 最能体现生产力的功能。

### 用户场景

```text
整理一下我关于压电薄膜谐振器的所有笔记，形成一个研究综述结构。
```

### 理想流程

Planner 自主决定：

1. 搜索相关文档；
2. 读取核心文档；
3. 整理主题结构；
4. 标出引用来源；
5. 生成综述草稿；
6. 提醒哪些资料重复、缺失或冲突。

### 推荐形态

先用 Skill 引导，不要写死 Workbench 流程。

可新增 Skill：

```text
主题整理助手
```

后续可配合写回草稿工具。

### 优先级

**P1 第三优先级**。付费感很强，但依赖 `read_docs`、引用体系和上下文先稳定。

---

## P1-4. 写回文档草稿

### 价值

问答停留在聊天框里，用户还要手动整理。能写回思源后，才形成真正闭环。

### 推荐工具

```text
create_doc_draft
append_to_doc_draft
insert_block_draft
preview_writeback
confirm_writeback
```

### 关键规则

* 默认只生成草稿或预览；
* 写入前必须用户确认；
* 不要悄悄修改原文；
* 写入类工具要标记 `requiresConfirmation: true`；
* 回答正文、整理结果、读书卡片、周报都可以写回。

### 优先级

**P1 第四优先级**。这是 VIP 付费转化的核心功能，但必须做好权限和确认。

---

## P1-5. 读书笔记深度整理

### 价值

非常适合你的项目背景，也非常适合思源用户。

### 功能方向

* 按书整理摘录；
* 区分摘录、评论、自己的思考；
* 生成章节级总结；
* 生成全书核心观点；
* 生成读书卡片；
* 生成主题卡片；
* 抽取行动清单；
* 对比两本书或两组笔记；
* 找出这本书和其他笔记的关联。

### 推荐形态

Skill + 现有知识库工具 + 后续写回草稿工具。

### 优先级

**P1 第五优先级**。很有付费价值，但最好在主题整理和写回能力稳定后做。

---

## P1-6. 本地文件问答

### 价值

非常适合作为 VIP 高级功能。用户可以临时选择 PDF、Word、Markdown、Excel、PPT、论文、产品手册、实验记录，然后让 AI 结合思源笔记回答。

### 推荐形态

独立文件资源工具：

```text
list_selected_files
read_file
search_file
summarize_file
```

### JSON 返回示例

```json
{
  "fileId": "file_xxx",
  "filename": "论文.pdf",
  "mimeType": "application/pdf",
  "content": "读取到的文本内容",
  "truncated": true,
  "nextCursor": "cursor_xxx"
}
```

### 关键规则

* 文件是临时资料；
* 思源知识库是长期知识；
* 回答可以同时引用思源文档和文件；
* references 中区分 `sourceType: "siyuan_doc"` 和 `sourceType: "file"`；
* 文件正文不应长期塞进历史上下文，只保留 references 元信息。

### 后移说明

本地文件问答涉及文件解析、chunk、引用和持久化，工程量较大，且需要 `read_docs`、引用体系和上下文先稳定。因此放到 P1-6，作为第一阶段最后一个高感知功能，而不是最优先项。

---

# 四、第二批高价值持续使用功能：P2

这些功能更适合形成长期留存和护城河。

## P2-1. 知识库体检

### 价值

这是长期留存功能。它让 Agent 不只是回答问题，而是每天帮用户维护知识库。

### 功能方向

* 哪些文档没有标签；
* 哪些文档标题太模糊；
* 哪些笔记长期没有反链；
* 哪些主题只有摘录没有总结；
* 哪些文档可能重复；
* 哪些新笔记未整理；
* 哪些主题已经形成体系；
* 哪些主题存在断点。

### 主页卡片示例

```text
今日知识库体检：
发现 8 篇未整理文档、3 个重复主题、2 个孤立主题、5 篇读书笔记缺少总结。
```

### 推荐形态

可以先做只读分析，不急着写回。

### 优先级

**P2 第一优先级**。非常有 VIP 价值，但需要更多结构分析工具。

---

## P2-2. 图片问答模式

### 价值

适合截图、论文图表、实验照片、软件报错、白板、手写笔记。

### 推荐形态

两种路线：

#### 路线 A：多模态 Provider

如果模型支持图片输入，Provider 直接接收图片资源。

#### 路线 B：图片资源工具

```text
analyze_image
describe_image
read_image_text
```

JSON 返回：

```json
{
  "imageId": "img_xxx",
  "description": "图片内容描述",
  "detectedText": "识别到的文字",
  "warnings": []
}
```

### 注意

* 不要默认 OCR；
* OCR 只作为 fallback；
* 图片资源也应进入 references；
* 历史上下文只保留 imageId / title / summary，不保存完整图片数据。

### 优先级

**P2 第二优先级**。有吸引力，但不是知识库 Agent 的第一核心。

---

## P2-3. 自定义 Skill UI

### 价值

高级用户会愿意为“我的 AI 工作流”付费。

### UI 应避免技术化

不要让用户写复杂 JSON。可以让用户填：

```text
Skill 名称：
适合场景：
回答风格：
注意事项：
不应该做什么：
```

### 示例 Skill

* 论文阅读助手；
* 读书笔记整理助手；
* 实验记录检查助手；
* 周报复盘助手；
* 项目管理助手；
* 写作润色助手。

### 优先级

**P2 第三优先级**。需要当前 Skill 系统和 UI 稳定后再做。

---

# 五、第三批平台化能力：P3

这些决定长期扩展性，但不是立刻最能拉动付费的功能。

## P3-1. MCP 服务接入

### 价值

让用户自己接外部工具和接口，扩展 Agent 能力。

### 可接入对象

* 日历；
* 邮件；
* GitHub；
* 网盘；
* 数据库；
* Zotero；
* 浏览器自动化；
* 自建 API；
* 实验室设备接口。

### 推荐架构

作为 Tool Provider 接入：

```text
Builtin Tools
Siyuan Tools
Web Tools
File Tools
Image Tools
MCP Tools
API Tools
```

MCP 工具进入 ToolRegistry 后，Planner 看到的是统一 ToolManifest：

```json
{
  "name": "mcp.xxx",
  "inputJsonSchema": {},
  "readOnly": true,
  "requiresConfirmation": false
}
```

### 风险

* 权限；
* token 泄露；
* 写操作安全；
* 工具过多导致 Planner 混乱；
* schema 质量不可控。

### 推荐节奏

先做只读 MCP 工具，后做写操作确认机制。

### 优先级

**P3 第一优先级**。长期很重要，但短期不如文件/联网/整理功能直接。

---

## P3-2. native tool-call provider

### 是什么

现在你的 Agent 是让模型输出：

```json
{
  "type": "tool",
  "toolName": "read_docs",
  "args": {}
}
```

native tool-call provider 是使用模型 API 原生 tools/function calling，让模型直接返回 provider-native tool call。

### 价值

* 工具调用更稳定；
* 参数更结构化；
* 更接近 Reasonix；
* 适合未来接更复杂工具生态。

### 当前建议

暂时不实现。你已经有 `native_tool_call` seam，足够未来扩展。当前先把 PromptJsonPlannerProvider 跑稳。

### 优先级

**P3 第二优先级**。

---

## P3-3. subagent / run_skill

### 是什么

Reasonix 里的 `run_skill` / subagent 是一种“让一个 Skill 开启子 Agent 任务”的能力。比如主 Agent 委托一个研究型子 Agent 单独搜索、整理、审查，最后只返回结果。

### 价值

适合：

* 深度研究；
* 长任务；
* 多步骤专题整理；
* 安全审查；
* 大规模知识库体检。

### 当前不建议实现的原因

* 容易让 Skill 绑定工具；
* 容易引入固定流程；
* 增加调试难度；
* 和当前第一铁律有潜在冲突；
* 需要任务隔离、结果回收、权限边界。

### 推荐

等 Skill 生态、写回、MCP、文件工具都稳定后再设计。

### 优先级

**P3 第三优先级**。

---

# 六、总优先级排名

## 第一阶段：基础稳定

1. **历史对话 JSON 上下文**
2. **读取工具命名为 `read_docs`**
3. **引用可点击、可复查、可继续读取**
4. **Debug 默认静默、单入口、有界内存**
5. **工具错误信息和 observation 展示继续打磨**

## 第二阶段：VIP 高感知能力

6. **日程 / 任务 / 日记综合 Skill**
7. **联网搜索**
8. **当前主题整理**
9. **写回文档草稿**
10. **读书笔记深度整理**
11. **本地文件问答**

## 第三阶段：长期留存能力

12. **知识库体检**
13. **图片问答模式**
14. **自定义 Skill UI**

## 第四阶段：平台化能力

15. **MCP 服务接入**
16. **native tool-call provider**
17. **subagent / run_skill**

# 七、我建议的近期开发路线

## 最近 1–2 个开发周期

先完成：

```text
conversationContext + read_docs + references 复查能力
```

目标是让 Agent Workbench 的连续对话和资料复用变稳。

## 接下来 2–4 个开发周期

做：

```text
日程任务日记 + 联网搜索
```

这两个最容易成为 VIP 感知功能，且日程/任务/日记可以优先复用现有思源知识库工具做只读查询。

## 再之后

做：

```text
主题整理 + 写回草稿 + 读书笔记整理
```

这三项开始真正把插件从“问答工具”升级成“知识生产工具”。

## 长期

做：

```text
本地文件问答 + 知识库体检 + MCP
```

形成长期使用习惯和平台化扩展能力。

# 八、VIP 付费吸引力总结

最能吸引用户付费的不是“AI 聊天”，而是：

1. **它懂我的思源知识库。**
2. **它能连续追问，不丢上下文。**
3. **它回答有真实引用，可以跳转复查。**
4. **它能结合本地文件、联网资料和我的笔记。**
5. **它能帮我整理主题、读书笔记和研究材料。**
6. **它能写回思源形成文档草稿。**
7. **它每天能帮我体检知识库和管理任务。**
