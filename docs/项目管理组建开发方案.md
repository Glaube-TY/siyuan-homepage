# 强化日记工作台：任务、记录、复盘一体化规划

## 一、当前最高原则

后续以这版为准：

1. **任务格式必须兼容现有 Tasks Plus 组件。**
2. **不使用自定义块属性作为主索引。**
3. **用 Markdown 标题层级判断内容区块。**
4. **任务、记录、项目推进、复盘内容尽量保存在思源原生日记文档中。**
5. **插件配置只保存规则和模板，不保存具体业务内容。**
6. **用户直接打开日记文档，也应该能看懂，不依赖隐藏属性。**
7. **新增任务、迁移任务、快速记录、项目推进，都必须写入明确标题区块，不允许随便写到文档末尾。**

也就是说，强化日记工作台不是一个单独数据库，而是：

> 一个基于思源原生日记文档、Markdown 标题结构、Tasks Plus 任务格式的个人工作台。

---

## 二、当前 Tasks Plus 任务格式确认

我看了当前项目里的 `tasksPlus` 组件，它现在的任务格式是：

```markdown
- [ ] 任务名称 ❗❗ ⌛2026-05-21 📅2026-05-24 🔁每周 ⏰2026-05-21 09:00 📍地点 #标签#
```

当前 Tasks Plus 已支持解析：

* `- [ ]` / `- [x]` / `- [X]`：完成状态；
* `❗` / `❗❗` / `❗❗❗` / `❗❗❗❗`：优先级；
* `⌛YYYY-MM-DD`：开始日期；
* `📅YYYY-MM-DD`：截止日期；
* `🔁重复规则`：重复；
* `⏰时间` 或 `⏰日期 时间`：提醒；
* `📍地点`：地点；
* `#标签#`：标签。

所以强化日记工作台新增任务时，必须生成这个格式，不重新发明任务格式。

---

## 三、任务格式扩展原则

Tasks Plus 的第一行任务格式不能随便加新符号，否则可能影响现有解析。

因此，后续扩展必须遵守：

```markdown
- [ ] 任务正文 ❗❗ ⌛2026-05-21 📅2026-05-24 #项目#
```

这一行继续保持 Tasks Plus 兼容。

如果需要附加迁移来源、说明、上下文，不放在第一行里，而是作为任务块下面的子内容，例如：

```markdown
- [ ] 完成强化日记工作台 ❗❗ ⌛2026-05-21 📅2026-05-24 #思源主页插件#
  - 迁移来源：((20250521000000-abcdefg "2025-05-21 日记"))
  - 迁移时间：13:25
```

这样有几个好处：

* 任务第一行完全兼容 Tasks Plus；
* 迁移来源用户能看懂；
* 不需要自定义属性；
* 不会破坏旧任务；
* 后续 Tasks Plus 可以选择性解析这些子内容，但旧逻辑不解析也不影响任务本身。

## 四、日记模板总结构

日记文档可能同时承载：

* 今日日记；
* 本周复盘；
* 本月总结；
* 年度总结。

因此一级标题固定为四个大区块。

```markdown
# 今日日记 {{date}}

# 本周复盘 {{week}}

# 本月总结 {{month}}

# 年度总结 {{year}}
```

插件判断时只看一级标题前缀：

* `# 今日日记`
* `# 本周复盘`
* `# 本月总结`
* `# 年度总结`

后面的日期、周数、月份、年份可以变化，但前缀不能变。

用户如果改了这些系统标题，插件不应该乱猜，而应该提示：

> 当前日记缺少「今日日记」区块，请补充模板或恢复标题。

## 五、今日日记区块结构

`# 今日日记` 是工作台核心区块。

建议默认模板如下：

```markdown
# 今日日记 {{date}}

{{完成标记}}

## 今日概览

- 今日重点：
- 今日状态：
- 今日一句话：

## 新建任务

> 今天新建的任务放在这里。任务格式兼容 Tasks Plus。

- [ ] 示例任务 ❗❗ ⌛{{date}} 📅{{date}} #示例#

## 迁移任务

> 从旧日记迁移到今天继续处理的任务放在这里。任务第一行保持原样，迁移来源写在任务下方。

## 快速记录

### 未分类

#### 13:01 记录

这里写具体记录内容。

### 想法

#### 13:05 记录

这里写想法内容。

### 问题

#### 13:10 记录

这里写问题内容。

### 决策

#### 13:15 记录

这里写决策内容。

### 日志

#### 13:20 记录

这里写过程日志。

## 项目推进

### 示例项目

- 今日进展：
- 遇到问题：
- 下一步：

## 任务动态

- 示例：新增任务 / 完成任务 / 删除任务 / 迁移任务。

## 今日复盘

### 我的感受

### 今日收获

### 遇到的问题

```

这里固定的二级标题是：

```markdown

## 新建任务
## 迁移任务
## 快速记录
## 项目推进
## 任务动态
## 今日复盘
```

这些标题就是插件和文档之间的结构协议。

## 六、新建任务与迁移任务

### 1. 新建任务

今天通过工作台新建的任务，写入：

```markdown
# 今日日记 {{date}}
## 新建任务
```

任务格式：

```markdown
- [ ] 任务名称 ❗❗ ⌛2026-05-21 📅2026-05-24 #项目#
```

新建任务的判断方式：

* 它在今天日记的 `## 新建任务` 区块内；
* 它是 Tasks Plus 格式任务项；
* 不需要额外写“创建日期”字段；
* 需要今日创建任务时，直接扫描今日日记的 `## 新建任务` 区块即可。

### 2. 迁移任务

有些任务是一年前创建的，但今天还在持续处理。如果还一直留在一年前的日记里，管理起来很不方便。

所以工作台需要提供一个操作：

> **迁移到今日日记**

迁移逻辑：

1. 用户在工作台看到一个旧任务；
2. 点击“迁移到今天”；
3. 插件找到今天日记；
4. 插件找到 `# 今日日记 -> ## 迁移任务`；
5. 把原任务移动到今天的 `## 迁移任务` 区块；
6. 原文档中的任务实体不再保留；
7. 迁移后的任务第一行保持和原来完全一致；
8. 在任务下方追加迁移来源。

迁移后的示例：

```markdown
## 迁移任务

- [ ] 完成强化日记工作台 ❗❗ ⌛2025-05-21 📅2026-05-24 #思源主页插件#
  - 迁移来源：((20250521000000-abcdefg "2025-05-21 日记"))
  - 迁移时间：13:25
```

这里的“迁移来源”建议链接到**来源日记文档**，不要链接到原任务块。

原因是：任务已经被移动或删除，原任务块可能不存在；但来源日记文档还在，反链更稳定。

技术上优先用 `moveBlock` 移动原任务块，而不是复制后删除。这样可以最大程度保留任务块原有内容。移动完成后，再给这个任务块追加子内容“迁移来源”。

如果 `moveBlock` 跨文档到指定位置的参数不确定，开发时需要让用户用 Apifox 测接口返回和行为，不要猜。

## 七、迁移提醒规则

工作台需要提醒用户迁移过旧任务。

建议增加一个配置项：

```text
任务迁移提醒天数：默认 30 天
```

判断规则：

* 任务未完成；
* 任务所在日记距离今天超过 N 天；
* 任务不在今天日记的 `## 新建任务` 或 `## 迁移任务` 中；
* 则在工作台中显示“建议迁移”。

示例提示：

> 该任务位于 2025-05-21 日记中，已超过 30 天未迁移。建议迁移到今日日记继续管理。

用户操作：

* `迁移到今天`；
* `暂不迁移`；
* 后续可以加“忽略本任务迁移提醒”，第一版可以不做。

注意：迁移提醒只影响展示，不自动迁移。

## 八、快速记录结构

你提得对，快速记录不应该全部用无序列表。那样后续编辑、删除、定位都不够清楚。

快速记录采用标题层级。

固定结构：

```markdown
## 快速记录

### 未分类

#### 13:01 记录

这里写记录内容。

### 想法

#### 13:05 记录

这里写想法内容。

### 问题

#### 13:10 记录

这里写问题内容。

### 决策

#### 13:15 记录

这里写决策内容。

### 日志

#### 13:20 记录

这里写过程日志。
```

规则：

* `## 快速记录` 是记录总区块；
* 三级标题是记录分类；
* 四级标题是具体记录；
* 四级标题默认用时间命名，例如 `#### 13:01 记录`；
* 正文写在四级标题下面，直到下一个四级、三级、二级或一级标题前。

默认分类：

```markdown
### 未分类
### 想法
### 问题
### 决策
### 日志
```

新增记录时：

1. 用户选择分类，默认“未分类”；
2. 插件找到 `## 快速记录`；
3. 找到对应三级分类；
4. 在该分类末尾插入：

   ```markdown
   #### 13:01 记录

   记录内容
   ```
5. 如果分类不存在，第一版不自动创建，提示补充模板或恢复标题。后续可以允许创建分类。

这样记录是纯 Markdown，结构清晰，用户直接读文档也能看懂。

## 九、项目推进结构

项目推进也用标题层级。

```markdown
## 项目推进

### 思源主页插件

- 今日进展：
- 遇到问题：
- 下一步：

### 论文

- 今日进展：
- 遇到问题：
- 下一步：
```

规则：

* `## 项目推进` 是项目推进总区块；
* 每个 `### 项目名` 是一个项目；
* 项目任务本身还是通过 Tasks Plus 标签聚合，例如 `#思源主页插件#`；
* 项目推进区只存自然语言推进记录；
* 不单独建项目数据库。

后续工作台项目区可以聚合：

* 标签里包含该项目名的未完成任务；
* 今天 `## 项目推进 -> ### 项目名` 下的推进记录；
* 近期快速记录中提到该项目的内容。

## 十、任务动态

任务动态用于保存重要操作日志，不作为任务实体。

```markdown
## 任务动态

- 新增任务：完成强化日记工作台
- 完成任务：修复逾期周期显示
- 迁移任务：完成论文仿真，从 2025-05-21 日记迁移到今天
- 删除任务：旧任务名称
```

任务动态可以用于删除任务时保留痕迹，也可以用于迁移任务时记录动作。

## 十一、删除任务操作

删除任务不能默认直接硬删，需要弹窗让用户选择。

删除任务弹窗：

标题：

```text
删除任务
```

说明：

```text
请选择删除方式。保留删除记录会在今日日记的「任务动态」中写入一条文本记录；直接删除会删除原任务块。
```

选项：

1. **保留删除记录**
2. **直接删除任务块**
3. **取消**

### 方式一：保留删除记录

流程：

1. 找到今天日记；
2. 找到 `# 今日日记 -> ## 任务动态`；
3. 写入：

   ```markdown
   - 删除任务：任务名称（来源：((来源日记文档ID "来源日记标题"))）
   ```
4. 删除原任务块。

这里仍然删除原任务块，只是保留一条文本日志。

### 方式二：直接删除任务块

流程：

1. 二次确认；
2. 调用思源删除块 API；
3. 不写任务动态。

### 为什么不做软删除标记

不建议使用：

```markdown
custom-ed-deleted="true"
```

因为我们已经决定不使用块属性作为主索引。软删除会让文档里长期留一堆无效任务，反而影响用户直接阅读。

## 十三、标题定位规则

插件通过 Markdown 标题层级定位内容。

规则：

```text
# 一级标题
从这个一级标题开始，到下一个一级标题之前，都属于这个大区块。

## 二级标题
从这个二级标题开始，到下一个同级或更高级标题之前，都属于这个小区块。

### 三级标题
从这个三级标题开始，到下一个同级或更高级标题之前，都属于三级区块。

#### 四级标题
从这个四级标题开始，到下一个同级或更高级标题之前，都属于一条具体记录。
```

示例：

```markdown
# 今日日记 2026-05-21

## 快速记录

### 想法

#### 13:01 记录

内容 A

#### 13:20 记录

内容 B

### 问题

#### 14:00 记录

内容 C

## 项目推进
```

插件解析结果应该是：

* `今日日记`

  * `快速记录`

    * `想法`

      * `13:01 记录`
      * `13:20 记录`
    * `问题`

      * `14:00 记录`
  * `项目推进`

如果用户改了系统标题，例如把 `## 快速记录` 改成 `## 灵感`，插件不要乱写，应提示：

> 当前日记缺少「快速记录」区块，请补充模板或恢复标题。

## 十四、CRUD 规划

### 1. 新增任务

落点：

```text
# 今日日记 -> ## 新建任务
```

流程：

1. 打开或创建今日日记；
2. 确认日记存在工作台模板；
3. 找到 `## 新建任务`；
4. 生成 Tasks Plus 格式任务；
5. 插入该区块末尾；
6. 刷新任务列表。

找不到 `## 新建任务` 时，不写入，提示补充模板。

### 2. 迁移任务

落点：

```text
# 今日日记 -> ## 迁移任务
```

流程：

1. 用户在工作台选择旧任务；
2. 判断旧任务所在日记日期是否超过迁移提醒阈值；
3. 点击“迁移到今天”；
4. 找到今天 `## 迁移任务`；
5. 优先用 `moveBlock` 移动原任务块；
6. 追加迁移来源子内容；
7. 刷新任务列表。

### 3. 查询任务

任务查询来源：

* 复用 Tasks Plus 的 `gettasksList()`；
* 复用或抽取 Tasks Plus 的 `formatTasksList()` 解析能力；
* 不重新写一套任务解析器。

工作台显示：

* 今日任务：开始日期或截止日期涉及今天；
* 逾期任务：截止日期早于今天且未完成；
* 今日新建：今天日记 `## 新建任务` 里的任务；
* 今日迁移：今天日记 `## 迁移任务` 里的任务；
* 建议迁移：任务所在日记超过 N 天且未完成。

### 4. 修改任务

复用 Tasks Plus 的任务编辑能力。

后续建议把 `tasksEditingDialog.svelte` 中的解析和生成逻辑抽出来，例如：

```text
tasksPlusParser.ts
```

提供：

* `parseTaskMarkdown()`
* `generateTaskMarkdown()`
* `extractTaskMeta()`

强化日记工作台不要复制这套逻辑。

### 5. 完成任务

直接更新任务块：

```markdown
- [ ] 任务
```

改成：

```markdown
- [x] 任务
```

这和 Tasks Plus 一致。

### 6. 删除任务

弹窗选择：

* 保留删除记录；
* 直接删除任务块；
* 取消。

删除时可以写入 `## 任务动态`。

### 7. 新增快速记录

落点：

```text
# 今日日记 -> ## 快速记录 -> ### 分类
```

插入内容：

```markdown
#### 13:01 记录

记录内容
```

### 8. 查询快速记录

第一版只查今天日记。

后续可以跨日期扫描所有日记里的：

```text
# 今日日记 -> ## 快速记录
```

### 9. 修改快速记录

直接 `updateBlock` 对应四级标题下的内容块。

如果记录由多个段落组成，需要定义：

* 四级标题是记录头；
* 从该四级标题到下一个四级或更高级标题前都是这条记录；
* 编辑时可以更新这个范围内的块。

第一版可以先只支持编辑记录正文第一块，后续再支持多块编辑。

### 10. 删除快速记录

弹窗选择：

* 直接删除该四级标题和其范围内内容；
* 或移动到 `## 任务动态` 记录一条“删除记录”。

第一版可以先做直接删除，删除前确认。

## 十五、Tasks Plus 需要做的兼容扩展

目前 Tasks Plus 已经能解析任务第一行，但为了工作台更好复用，建议后续对 Tasks Plus 做**兼容式增强**，不是破坏式改造。

### 1. 抽离任务解析器

当前解析逻辑在 `tasksPlus.ts` 和 `tasksEditingDialog.svelte` 里都有。

建议抽出：

```text
components/utils/widgetBlock/widget/tasksPlus/tasksPlusParser.ts
```

导出：

```ts
parseTaskLine(markdown: string): ParsedTask
generateTaskLine(data: ParsedTask): string
isTaskCompleted(taskCheck: string): boolean
extractTaskTags(markdown: string): string[]
```

Tasks Plus 和强化日记工作台都用它。

### 2. 保持第一行兼容

不要新增会影响解析的新符号。

不建议在任务第一行增加：

```text
🔗来源
🧭迁移
🗂项目
```

这些新符号可能会影响现有 `split(/[📅⌛❗🔁⏰📍#]/)` 或正则。

如果要表达迁移来源，用子内容。

### 3. 增加迁移来源读取

工作台可以额外读取任务块子内容：

```markdown
- 迁移来源：((来源日记ID "来源日记"))
```

但这个是增强信息，不影响 Tasks Plus 原本列表。

### 4. 支持任务所在日记判断

工作台需要知道任务在哪个日记中。

可以通过任务块的：

* `root_id`；
* `hpath`；
* 所在文档是否有 `custom-dailynote-yyyymmdd`；
* 或文档标题日期；

来判断任务属于哪天日记。

不确定 API 返回时，先让用户用 Apifox 测，不猜。

# 给新 AI 的续接摘要

当前正在开发的是思源笔记主页插件中的“强化日记”功能。它已经完成了日记、周复盘、月总结、年度总结的 MVP，支持基于思源原生日记文档的模板补充、完成、取消完成、跳过、取消跳过。现在准备升级成“任务 + 记录 + 复盘工作台”。

后续方案以以下规则为准：

1. 数据尽量保存在思源原生日记文档中。
2. 不使用自定义块属性作为主索引。
3. 使用 Markdown 标题层级定位内容区块。
4. 任务格式必须兼容现有 Tasks Plus 组件。
5. Tasks Plus 任务格式为：
   `- [ ] 任务名称 ❗❗ ⌛YYYY-MM-DD 📅YYYY-MM-DD 🔁重复 ⏰提醒 📍地点 #标签#`
6. 任务第一行不能随便增加新符号，避免破坏 Tasks Plus 解析。
7. 任务迁移来源等扩展信息放在任务块子内容中，不放在任务第一行。
8. 日记一级标题固定为：

   * `# 今日日记 {{date}}`
   * `# 本周复盘 {{week}}`
   * `# 本月总结 {{month}}`
   * `# 年度总结 {{year}}`
9. 今日日记二级标题固定为：

   * `## 今日概览`
   * `## 新建任务`
   * `## 迁移任务`
   * `## 快速记录`
   * `## 项目推进`
   * `## 任务动态`
   * `## 今日复盘`
10. 新建任务写入 `# 今日日记 -> ## 新建任务`。
11. 迁移任务写入 `# 今日日记 -> ## 迁移任务`。
12. 迁移任务保持原任务第一行完全不变，只在任务下方追加：
    `- 迁移来源：((来源日记文档ID "来源日记标题"))`
13. 旧任务如果所在日记超过默认 30 天且未完成，工作台提示用户迁移到今天。
14. 快速记录使用标题结构，不用无序列表作为每条记录：

    * `## 快速记录`
    * `### 分类`
    * `#### 13:01 记录`
15. 默认记录分类为：未分类、想法、问题、决策、日志。
16. 项目推进结构为：

    * `## 项目推进`
    * `### 项目名`
17. 今日创建任务直接通过今天日记的 `## 新建任务` 判断。
18. 今日迁移任务直接通过今天日记的 `## 迁移任务` 判断。
19. 删除任务时弹窗让用户选择：

    * 保留删除记录；
    * 直接删除任务块；
    * 取消。
20. 保留删除记录时，在今天 `## 任务动态` 写入：
    `- 删除任务：任务名称（来源：((来源日记文档ID "来源日记标题"))）`
    然后删除原任务块。
21. 后续要抽离 Tasks Plus 解析逻辑，避免强化日记复制一套任务解析器。
22. 不确定思源 API 返回结构时，必须先让用户用 Apifox 测，不要猜。
23. 不要动 `advanced.ts`，不要改会员激活逻辑。
24. 不要直接做大 UI，后续开发顺序是：
    模板升级 → 标题解析 → 区块定位 → 新建任务 → 迁移任务 → 快速记录 → 工作台 UI → 完整页面。
